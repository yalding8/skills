# Verify Blockers

在创建 Issue 或标记任务为"阻塞/待外部输入"之前，先自动验证假设是否成立。避免把已解决的问题当作阻塞项跟踪。

## 触发时机

- 用户说"这个需要等 XX"或"这个依赖 XX"时
- 创建新 Issue 时发现有 Blocked by 字段
- Sprint Plan 评审时扫描所有"⏳ 待"状态项
- 用户主动调用 `/verify-blockers`

## 验证流程

### 1. 收集假设

扫描以下来源，提取所有被标记为"阻塞/待处理/依赖外部"的项：
- GitHub Open Issues（Blocked by 字段）
- Sprint Plan 中的"⏳"状态项
- 用户对话中提到的"需要 XX 才能做"

### 2. 分类验证

对每个假设的阻塞项，按类型尝试自动验证：

**网络/服务类**（10 秒可验证）：
- DNS 记录 → `nslookup` 检查（SPF/DKIM/DMARC/MX）
- API 连通性 → `curl` 测试端点
- 服务状态 → `docker ps`、`pg_isready`、`redis-cli ping`
- 端口可达 → `nc -z host port`

**数据可用性类**（30 秒可验证）：
- 邮件附件 → Graph API 检查收件箱是否已有
- 文件存在 → `ls`/`find` 检查本地是否已有
- API Key → 环境变量检查 + 简单 API 调用验证
- 数据库状态 → SQL 查询

**配置/权限类**（1 分钟可验证）：
- 邮箱权限 → SMTP/IMAP 登录测试
- OAuth2 → token 获取测试
- Feature flag → 配置文件检查

**真正的外部依赖**（无法自动验证）：
- 法务评审 → 标记为"真阻塞"，不尝试验证
- 业务决策 → 标记为"真阻塞"
- 第三方审批流程 → 标记为"真阻塞"

### 3. 输出报告

对每个阻塞项输出：

```
| 阻塞项 | 验证方式 | 结果 | 行动 |
|:-------|:---------|:-----|:-----|
| SPF/DKIM 配置 | nslookup -type=TXT | ✅ 已配置 | 关闭 Issue |
| EWS 连通性 | exchangelib connect | ❌ Basic Auth 被禁 | 改用 SMTP+Graph |
| Excel 样本 | Graph API 检查附件 | ✅ 收件箱已有 3 个 | 直接下载使用 |
| 法务评估 | 无法自动验证 | ⏳ 真阻塞 | 保持跟踪 |
```

### 4. 核心原则

> **"30 秒规则"**：如果一个阻塞项可以在 30 秒内通过一条命令验证，那就先验证再标记。
> 不要把"我不确定"当作"它一定不行"。
> 假设的成本远高于验证的成本。

## 反模式清单

以下是常见的"假阻塞"模式，看到时应立即验证：

- "需要 IT 配置 XX" → 先查 DNS/端口，可能已经配好了
- "需要 XX 提供数据" → 先查现有数据源（邮箱、数据库、文件），可能已经有了
- "XX 服务不可用" → 先试一下，可能只是认证方式不对
- "需要申请 XX 权限" → 先用现有凭证试一下，可能已经有权限
- "等 XX 版本发布" → 先查当前版本是否已支持该功能
